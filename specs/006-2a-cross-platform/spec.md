## Overview

005 で確立した patterns/ パターン集（同一契約を 3 フレームワークで実装比較）に、
Anthropic タクソノミーの残り4パターンを追加してコアタクソノミーを全網羅する。
あわせて、パターンが 2→6 に増えることで顕在化する契約複製コスト（最大
6×3=18 コピー）を、依存ゼロの契約専用パッケージ `patterns/contracts/` への
**shared-contracts 昇格**で解消する。

本イテレーションの価値は2点。第一に、prompt-chaining / parallelization /
evaluator-optimizer / autonomous-agent を 3 フレームワークで横並びに比較できる
ようにすること。とりわけ唯一の「Agent」型である autonomous-agent は OWASP
Agentic AI Top 10 の主戦場であり、ガードレール4種（最大反復数・ツール許可
リスト・ヒューマン承認フック・ループ毎予算記録）を**契約レベルで全レーン共通化**
することで、多層防御の参照実装を提供する。第二に、契約の正本（README）と
実装（パッケージ）の一致を保証する仕組みを、18 コピーのクロス複製ドリフト検知
から「README 正本 == パッケージ」の1点へ縮約し、保守コストを抑えること。

## Glossary

| 用語 | 定義 |
|------|------|
| レーン (lane) | `patterns/frameworks/<fw>/` 配下の独立 uv プロジェクト1つ。 |
| パターン契約 (pattern contract) | 全レーンが共有する入出力 Pydantic モデルとエントリポイント関数シグネチャ。本イテレーションから正本は `patterns/<pattern>/README.md`、実体は `patterns/contracts/` パッケージ。 |
| shared-contracts | `patterns/contracts/` に新設する契約専用パッケージ。フレームワーク SDK に非依存（pydantic のみ）、`requires-python >=3.13`。各レーンが `tool.uv.sources` のパス依存で import する。 |
| prompt-chaining | タスクを逐次ステップに分解し、各ステップ間にプログラム検証ゲートを置く。ゲート不合格なら早期終了するワークフロー（Anthropic taxonomy）。 |
| parallelization | 同一タスクを並列に fan-out するワークフロー。sectioning（独立サブタスク分割）と voting（同一タスク多数決）の2変種を持つ（Anthropic taxonomy）。 |
| evaluator-optimizer | 生成器が候補を作り評価器が判定・フィードバックするループ。判定が pass か上限到達で打ち切る（Anthropic taxonomy）。 |

| ガードレール (guardrail) | autonomous-agent の安全機構: 最大反復数 / ツール許可リスト / 危険操作のヒューマン承認フック / ループ毎予算消費記録（予算はトークン数 `int`）の4種。 |

### Requirement 1: shared-contracts パッケージ昇格

パターンが 6 に増えることで顕在化する契約複製コストを、契約専用パッケージへの
集約で解消する。

**Acceptance Criteria**

1.1 THE システム SHALL `patterns/contracts/` に契約専用 uv パッケージを新設し、
そのパッケージはフレームワーク SDK（pydantic-ai / beeai-framework /
llama-index）に依存せず pydantic のみを runtime 依存とする。

1.2 THE `patterns/contracts/` パッケージ SHALL `requires-python >=3.13` を
宣言し、`>=3.14` レーン（pydantic-ai）と `>=3.13,<3.14` レーン（beeai /
llamaindex）の両方からインストール可能であること。

1.3 THE システム SHALL routing / orchestrator-workers / prompt-chaining /
parallelization / evaluator-optimizer / autonomous-agent の6パターンの契約
（入出力 Pydantic モデルと `Literal` 経路語彙）を `patterns/contracts/`
パッケージに集約する。

1.4 各レーン SHALL `tool.uv.sources` のパス依存で `patterns/contracts/` を
import し、レーン内に契約モデルの複製（旧 `contracts.py` のクロス複製）を
持たないこと。

1.5 THE システム SHALL 既存の routing / orchestrator-workers 契約を本パッケージへ
移行し、移行後も両パターンの3レーン実装が同一契約シグネチャで動作すること。

### Requirement 2: 契約ドリフト検知の単一点化

契約の正本（README）と実体（パッケージ）の一致を1点で保証する。

**Acceptance Criteria**

2.1 THE システム SHALL 各パターン README に記載した契約（正本）と
`patterns/contracts/` パッケージの定義が一致することを検証するドリフトテストを
1点で提供する。

2.2 THE ドリフトテスト SHALL `patterns/contracts/` パッケージのみを参照し、
レーン間のクロス複製比較（旧 `test_patterns_contract_sync.py` の3レーン AST
比較）に依存しないこと。

2.3 IF README 正本とパッケージ定義のクラス集合またはフィールド集合が乖離した
場合、THEN THE ドリフトテスト SHALL 失敗すること。

### Requirement 3: prompt-chaining パターン

逐次ステップ + ステップ間ゲートのワークフローを3フレームワークで実装する。

**Acceptance Criteria**

3.1 パターン契約は SHALL 次の通り: 型付き中間成果の連鎖
`ChainStep{name: str, output: str}` のリストとゲート判定
`GateOutcome{passed: bool, detail: str}` を持つ
`ChainResult{steps, gate, final_output: str | None}`、エントリポイント
`async def run_prompt_chain(input_text: str, *, model/llm) -> ChainResult`。

3.2 各レーン SHALL ステップを逐次連結し、各ステップ出力を次ステップの入力と
する: PydanticAI = 複数 `agent.run` の逐次連結、LlamaIndex = `@step` の直列連鎖、
BeeAI = Workflow（Pydantic state）の逐次ステップ。

3.3 IF ステップ間ゲート（プログラム検証）が不合格となった場合、THEN THE 実装
SHALL チェーンを早期終了し、`final_output` を `None` とし、不合格のゲート判定が
`ChainResult` から判別可能であること（silent な継続は禁止）。

### Requirement 4: parallelization パターン

並列 fan-out を sectioning / voting の2変種で実装する。

**Acceptance Criteria**

4.1 パターン契約は SHALL 次の通り:
`run_parallelization(task: str, *, variant: Literal["sectioning","voting"],
model/llm, n: int = 3) -> ParallelResult`、出力
`ParallelResult{variant, branches: list[Branch], aggregate: str}`、
`Branch{index: int, output: str}`。

4.2 WHERE `variant == "sectioning"`、THE 実装 SHALL タスクを独立サブタスクに
分割して並列実行し、各ブランチ出力を集約した `aggregate` を返す。

4.3 WHERE `variant == "voting"`、THE 実装 SHALL 同一タスクを `n` 回並列実行し、
多数決による `aggregate` を返す。

4.4 ワーカー実行は SHALL 並列（`asyncio.gather` または各フレームワークの
fan-out 機構）で行い、`branches` の順序は決定論的に復元されること
（LlamaIndex の `collect_events` 等で順序が乱れないこと）。

### Requirement 5: evaluator-optimizer パターン

生成器と評価器のループを3フレームワークで実装する。

**Acceptance Criteria**

5.1 パターン契約は SHALL 次の通り:
`run_evaluator_optimizer(task: str, *, model/llm, max_iterations: int = 3) ->
OptimizationResult`、出力
`OptimizationResult{iterations: list[Iteration], final_output: str,
stop_reason: Literal["passed","max_iterations"]}`、
`Iteration{index: int, candidate: str, verdict: Literal["pass","revise"],
feedback: str}`。

5.2 THE 実装 SHALL 評価器の判定が `pass` になるか `max_iterations` に到達する
まで生成→評価のループを反復し、各反復を `iterations` に記録する。

5.3 WHEN 評価器が `revise` を返した場合、THE 実装 SHALL そのフィードバックを
次反復の生成器入力に反映する。

5.4 THE 実装 SHALL ループ打ち切りの理由を `stop_reason` に記録し、`passed` /
`max_iterations` 以外の値を取らないこと（語彙は `Literal` で固定）。

### Requirement 6: autonomous-agent パターン（ガードレール契約）

ツールループ + 環境フィードバック + 停止条件の自律エージェントを、
ガードレール4種を契約レベルで共通化して実装する。

**Acceptance Criteria**

6.1 パターン契約は SHALL 次のエントリポイントを全3レーンで同一シグネチャで
公開する: `run_autonomous_agent(goal: str, *, model/llm, max_iterations: int,
allowed_tools, approval_hook, budget: int) -> AgentRunResult`、出力
`AgentRunResult{steps: list[AgentStep], final_output: str | None,
stop_reason, total_budget_spent: int}`、
`AgentStep{index: int, tool: str, observation: str, budget_spent: int}`。
`budget` / `budget_spent` / `total_budget_spent` は SHALL トークン数を表す
非負整数（`int`）とする（コスト換算は将来イテレーションに委ねる。OWASP
「無制限消費」はトークン主戦場であり、3フレームワークとも usage/token を
露出するため移植性と決定論性が高い）。

6.2 THE `stop_reason` SHALL `Literal["completed","max_iterations",
"budget_exceeded","denied"]` で語彙固定とする。

6.3 WHEN ツールループの反復数が `max_iterations` に到達した場合、THE 実装
SHALL ループを停止し `stop_reason` を `max_iterations` とする。

6.4 IF エージェントが `allowed_tools` に含まれないツールを呼び出そうとした
場合、THEN THE 実装 SHALL そのツールを実行せず拒否すること。

6.5 WHEN 危険操作に分類されたツール呼び出しが発生した場合、THE 実装 SHALL
`approval_hook` を介して承認を求め、承認が得られない場合はループを停止し
`stop_reason` を `denied` とする。

6.6 THE 実装 SHALL 各反復のループ予算消費（トークン数）を
`AgentStep.budget_spent` に記録し、累積（`total_budget_spent`）が `budget`
を超過した場合はループを停止して `stop_reason` を `budget_exceeded` とする。

### Requirement 7: オフラインテスト

新4パターンを完全 hermetic に検証する。autonomous-agent のツールループは
台本化フェイクで決定論化する。

**Acceptance Criteria**

7.1 各レーンのユニットテストは SHALL ネットワーク I/O ゼロで実行可能であること。
フェイクは 005 を踏襲: pydantic-ai = TestModel / FunctionModel、
beeai = `ChatModel` 継承の ScriptedChatModel、llamaindex = `CustomLLM` 継承
フェイク。

7.2 autonomous-agent のオフラインテストにおいて、各レーンのフェイク SHALL
「ツール呼び出し → 環境フィードバック → 最終回答」のターン列を台本として返し、
ツールは決定論的なインメモリスタブとすること（各反復で同一の停止経路を再現）。

7.3 各レーンは新4パターンそれぞれについて SHALL 正常系 + 契約違反系
（prompt-chaining ゲート不合格 / autonomous-agent のツール許可リスト違反・
予算超過・承認拒否 / evaluator-optimizer の max_iterations 打ち切り）の
ユニットテストを持つこと。

7.4 各レーンのカバレッジは SHALL `fail_under`（005 で確立したレーン毎フロア、
ratchet で引き上げ）を満たすこと。

7.5 各レーンのテストは SHALL レーンディレクトリ内で `uv run pytest` により
独立実行できること。

### Requirement 8: Ollama 結合テスト

**Acceptance Criteria**

8.1 各レーンは SHALL 新4パターンについて `RUN_INTEGRATION_PATTERNS=1`
環境変数でゲートされた結合テストを `tests/integration/` に持つ（未設定時 skip）。

8.2 結合テストのアサーションは SHALL 契約レベルに留めること（prompt-chaining =
ステップが1件以上 / parallelization = branches が `n` 件かつ aggregate 非空 /
evaluator-optimizer = stop_reason が語彙内 / autonomous-agent = stop_reason が
語彙内。正確なテキスト一致は禁止）。

8.3 結合テストは SHALL `OLLAMA_BASE_URL` / `OLLAMA_MODEL_NAME` を環境変数から
読むこと。
