## Investigations

### Live reference for the V2 (Beta) `Model` ABC

- **Question**: What are the exact V2 method signatures `LiteLLMModel` must implement?
- **Findings**: `WatsonxSDKModel` is the authoritative in-repo reference — same
  ABC, same project conventions. `request` is 3-arg; `request_stream` is an
  `@asynccontextmanager` returning `AsyncGenerator[StreamedResponse]` and may
  raise before its unreachable `yield`. `system`/`model_name` are properties.
- **Evidence**: [watsonx.py:267-578](../../src/pydantic_ai_sandbox/llm/providers/watsonx.py#L267-L578).

### Current `_build_litellm` and the 404 root cause

- **Question**: Why is the existing litellm transport broken, and what exactly is replaced?
- **Findings**: It builds `OpenAIChatModel` over `LiteLLMProvider` (a thin
  `AsyncOpenAI` wrapper) which POSTs to `/chat/completions` — not exposed by
  watsonx.ai → 404 (confirmed in `002` live verification, see memory
  `watsonx-integration-findings`). Routing via `litellm.acompletion()` lets
  LiteLLM perform the watsonx-specific request translation.
- **Evidence**: [watsonx.py:581-668](../../src/pydantic_ai_sandbox/llm/providers/watsonx.py#L581-L668).

### `litellm.acompletion()` return shape vs the shared `build_response(dict)` contract

- **Question**: C1's `build_response` was generalised from watsonx `achat`, which
  returns a bare `dict`. What does `litellm.acompletion()` return, and does it
  satisfy that `dict` contract?
- **Findings**: `acompletion()` returns a `litellm.ModelResponse` **object**, not
  a plain `dict`. C2 must normalise it via `.model_dump()` before handing to C1.
  `.model_dump()` preserves `tool_calls[].function.arguments` as the **raw JSON
  string** the backend emitted — exactly what Req 2.4 requires (Granite
  double-encoded args surfaced faithfully, never re-parsed/re-encoded). Variants
  that re-parse arguments into a nested object would corrupt Req 2.4 and are
  rejected.
- **Consequence**: C2 owns this one-line normalization; `build_response` keeps its
  `dict` signature unchanged, so Req 11.4 (watsonx behavior unchanged) holds.
- **Evidence**: [watsonx.py:490-546](../../src/pydantic_ai_sandbox/llm/providers/watsonx.py#L490-L546)
  (the `dict`-consuming reference); confirm exact `.model_dump()` arg-string
  preservation in the response-mapping unit test (Req 10.1).

### Optional dependency wiring

- **Question**: Is `litellm` already an optional extra and dev dependency?
- **Findings**: Yes — `[project.optional-dependencies] litellm = ["litellm"]`
  plus a dev-group entry so the hermetic suite can import it. No `pyproject`
  change needed.
- **Evidence**: `pyproject.toml:25-33`.

## Architecture decisions

### ADR-1: Extract shared mapping into `llm/_openai_mapping.py`

- **Context**: Both `WatsonxSDKModel` and `LiteLLMModel` need identical
  OpenAI-shape mapping; duplicating it violates DRY and risks drift.
- **Decision**: Move the `_map_*` helpers + `_FINISH_REASON_MAP` to a new
  `llm/_openai_mapping.py`; generalise `_build_response` into a free
  `build_response(raw, *, model_name, provider_name)`. `watsonx.py` imports them
  (re-exporting via `__all__` with `# pyright: ignore[reportPrivateUsage]` per
  the cross-module underscore convention).
- **Alternatives**: Keep in `watsonx.py` and import from litellm (couples a
  provider to another) — rejected; base class with mapping methods (inheritance
  complicates testing) — rejected.
- **Consequences**: Req 11.4 must hold — `WatsonxSDKModel` behavior is
  byte-for-byte unchanged; existing watsonx tests are the regression net.

### ADR-2: Route via `acompletion()` with `num_retries=0`

- **Context**: The OpenAI-client path 404s; `FallbackModel` must be the only
  resilience layer (project ADR-2 / tech.md).
- **Decision**: Call `litellm.acompletion(..., num_retries=0)`; wrap every
  exception as `ModelAPIError`. Mapping/response errors stay unwrapped (fail loud).
- **Alternatives**: sync `litellm.completion()` + `to_thread` (latency) — rejected;
  enable LiteLLM retries (double resilience) — rejected.
- **Consequences**: Broad `except Exception` is intentional (Req 4.1); ruff
  `BLE001` may need a scoped `# noqa` with rationale.

### ADR-3: watsonx project ID via `WATSONX_PROJECT_ID` env var

- **Context**: Req 7.2 mandates routing the project ID to LiteLLM **exclusively**
  via the `WATSONX_PROJECT_ID` env var, failing fast if missing. But
  `Settings.watsonx_project_id` already reads that env var, and the boot-time
  credential gate already requires it whenever watsonx is selected
  (`config.py:236-244`). LiteLLM, however, reads `os.environ["WATSONX_PROJECT_ID"]`
  **directly** — not via `Settings`.
- **Decision**: The builder reconciles the two: it reads the validated
  `settings.watsonx_project_id` and ensures it is present in `os.environ` for
  LiteLLM (e.g. set it from settings if absent), failing fast with a clear message
  if the value is `None`. This keeps a single source of truth (`Settings`) while
  satisfying LiteLLM's env-var contract.
- **Alternatives**: Pass project ID as an `acompletion` arg — rejected (LiteLLM's
  watsonx path expects the env var); trust the deployment to export it — rejected
  (pydantic-settings may load it from `.env` without exporting to `os.environ`,
  the exact silent-404 class this feature exists to kill).
- **Consequences**: ⚠️ See risk below — the builder must not double-validate in a
  way that diverges from the credential gate's error message. Writing to
  `os.environ` is a **process-global side effect** in an otherwise pure, I/O-free
  builder (against tech.md's no-side-effect construction convention); accepted
  only because LiteLLM reads the env var directly. Hermetic tests for this branch
  MUST use `monkeypatch.setenv`/`delenv` (or save-and-restore) so the write does
  not leak across tests. If the live lane (Req 10.3) proves an
  `acompletion(project_id=...)` kwarg routes correctly, prefer that and drop the
  env write entirely.

## Risks & open questions

- ⚠️ **`.env`-loaded project ID not in `os.environ`** — pydantic-settings can
  populate `settings.watsonx_project_id` from a `.env` file without exporting it
  to the process environment, so LiteLLM's direct `os.environ` read would miss
  it and 404. *Mitigation*: ADR-3 — the builder sets the env var from validated
  settings before constructing the model. Verify with a hermetic test that asserts
  `os.environ` is populated **and uses `monkeypatch` so the write is restored**
  after the test (no cross-test leakage).
- ⚠️ **Broad `except Exception` vs ruff `S`/`BLE` lanes** — Req 4.1 mandates it;
  the security/lint lane must stay green. *Mitigation*: scoped `# noqa: BLE001`
  with a docstring citing Req 4.1.
- ✅ **LiteLLM `timeout` argument shape** *(resolved)* — `acompletion` accepts
  `timeout: float | str | httpx.Timeout`, so the SDK path's
  `httpx.Timeout(read, connect=connect)` is passed through directly and **both**
  connect and read reach the backend (Req 5.1). The earlier single-float
  fallback is dropped — it would have silently discarded the connect phase and
  violated Req 5.1's "connect and read".
- ❓ **`StreamedResponse` import for the `request_stream` signature** — mirror
  `watsonx.py`'s `TYPE_CHECKING` import; confirm the type still exists in
  `2.0.0b6`. *Resolve in*: Task implementation (the reference compiles today).
- ❓ **`custom_llm_provider` necessity for watsonx** — the `watsonx/<id>` route
  prefix may already select the backend, making `custom_llm_provider` redundant
  for C3. *Resolve in*: live integration lane (Req 10.3).
